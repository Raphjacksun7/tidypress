TidyPress starts from a boring premise that still needs saying in 2026: **the product is the tree you publish from.** Markdown on disk. A small `tidypress.config.ts`. A build step that turns intent into bytes. A home page that foregrounds *your* writing and *your* projects — not a generic docs portal that accidentally becomes the whole company.

I am an engineer. My public work is repos, shipped tools, essays, and reference pages. I want one owned URL that LinkedIn, GitHub, and the next model citation can point to — current, legible, and under my control.

## Voices worth stealing (with links)

[Tim Berners-Lee](https://www.w3.org/History/1989/proposal.html) framed the web as a **web of documents and links** with real URLs — still the right ancestor for a public engineer surface.

[Tom Preston-Werner](https://tom.preston-werner.com/2008/11/17/blogging-like-a-hacker) named the workflow TidyPress extends: **writing in git**, static output, publish from the repo (the post that launched Jekyll).

[Brandur Leach](https://brandur.org/fragments/graceful-degradation-time) practices it today: **Markdown + Git**, public source on GitHub, [publish on push](https://brandur.org/fragments/write-publish-pipeline), fixes via pull request — a blog built to outlive any CMS.

[Donald Knuth](https://www-cs-faculty.stanford.edu/~knuth/lp.html) called the deeper habit **literate programming**: treat the human-readable source as the artifact; the rendered page is downstream.

[Git’s snapshot model](https://git-scm.com/book/en/v2/Getting-Started-What-is-Git%3F#_snapshots_not_differences) and [review on pull requests](https://docs.github.com/en/get-started/using-github/github-flow) already treat text like code when teams care. I want the same habit for a personal site: prose in the PR, HTML in `build/`.

## The layout TidyPress assumes

Not every site needs the same shape. The defaults bias toward **engineer authorship**: ideas and work on the home page; dated writing in `writing/`; optional `docs/` when something actually needs a manual.

<FileTree>
- docs/
  - **tidypress.config.ts**
  - tidypress-context.md
  - src/
    - content/
      - writing/
      - projects/
      - docs/
  - build/
    - index.html
    - pagefind/
</FileTree>

`tidypress context` exports a markdown snapshot of titles and paths — useful memory for you *and* for an agent with repo access, without handing either one a hosted draft database.

## Build as receipt

The build step matters the way compiler output matters: intent becomes an artifact you can point at. `tidypress build` gives you HTML and a [Pagefind](https://pagefind.app/docs/) index at build time — what search indexes is what you built, not a parallel copy maintained elsewhere.

<Mermaid align="center" code={`
flowchart LR
  subgraph repo [Git repo]
    W[writing/]
    P[projects/]
    D[docs/]
  end
  repo --> Config[tidypress.config.ts]
  Config --> Build[tidypress build]
  Build --> Out[build/]
  Out --> Host[Your static host]
`} />

```bash
git diff main...HEAD -- docs/src/content/
tidypress build
# ship docs/build/ — same artifact CI built
```

<Image src="/images/writing/print-output.gif" alt="Sheets feeding through a printer" caption="A build is a print run: fixed pages you can ship, not a live feed to babysit." />

That rhythm — pre-render once, serve files — is what [Jamstack](https://jamstack.org/) names; the [Web Almanac](https://almanac.httparchive.org/en/2022/jamstack) still describes it as fast pages from pre-rendering and caching. I want that predictability for text-heavy, engineer-owned sites.

### Embed your own ship artifact

Static output is HTML on disk. You can **iframe your own `build/` demo** inside a post or readme — same host, same bytes users get in production. (Third-party sites often block embedding; a file under `docs/public/` does not.)

</div>

<Callout type="tip">
MDX accepts raw `<iframe>`. Point at something you control under `docs/public/` so the frame always loads.
</Callout>

## Docs without becoming a docs company

People sometimes ask if TidyPress is “a documentation product.” It can hold docs. I use it when the site is **mostly** essays and work on the home page, with reference pages in `docs/` when a library or tool needs them. You should not have to dress as a wiki to get a calm reading column.

<Callout type="quote">
Own the tree. Let the build prove what shipped. Keep the reading experience quiet enough to finish a post.
</Callout>

## So yes

For me, the folder is the product. Editors, agents, and CI stay upstream. What remains after save — and what `build/` contains — is the site. TidyPress is the thin layer that makes that habit feel obvious instead of heroic.
