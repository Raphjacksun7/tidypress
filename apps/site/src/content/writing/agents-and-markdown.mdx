The interesting shift with coding agents is not that publishing got easier. It is that **authorship got cheaper.** You still need a point of view, a public URL, and a build step you trust. The agent did not remove those. It removed the friction between “I should write this down” and a file diff.

TidyPress is built for that world — but it is not built *for* agents. There is no chat surface, no draft host, no sync API. The site is whatever is in the tree after you merge. That is the same contract vim users had twenty years ago, and it is the right contract for agents too: **they propose edits; you keep the repo.**

## The public interface stays human-readable

Followers, hiring managers, and future collaborators do not need another dashboard — they need a **stable presentation layer**: writing that ages, projects that showcase work, docs when a tool needs a manual. Agents can help fill the tree; TidyPress turns the tree into HTML you own.

[Simon Willison](https://simonwillison.net/) documents the engineering side in [Using Git with coding agents](https://simonwillison.net/guides/agentic-engineering-patterns/using-git-with-coding-agents/): agents fluent in commits, branches, and PRs; you keep version control as the record. That maps onto a TidyPress site when **markdown in git** is canonical after merge — same shape as [Brandur Leach’s write → publish pipeline](https://brandur.org/fragments/write-publish-pipeline), but with `tidypress build` as the receipt.

Willison draws a harder line for **opinionated prose**: “I” stays human-written ([his policy](https://simonwillison.net/2026/Mar/1/ai-writing/)). TidyPress does not pick that fight — you decide what lands in `writing/` — but the tool assumes the draft is always a file in the repo, never a hosted editor’s database. `tidypress context` gives an agent the map of the tree; it does not get custody.

## The tree both of you edit

Platforms that own your drafts eventually own your audience. A folder of markdown plus a static `build/` output is boring on purpose. Boring survives model upgrades, editor churn, and the next hot publishing startup.

Presets (`lab`, `blog`, `persona`) exist so the tree already looks like a **public engineer site** — work and writing on the home page — instead of a generic `content/` dump. Before a large refactor, `tidypress context` gives the model structured memory **without** giving it custody.

## Same pipeline, different keyboard

What changes when Cursor or Claude writes half the posts? Almost nothing operationally.

| Step | Human | Agent |
|------|--------|--------|
| Draft | Type in `writing/` | Open PR with new `.mdx` |
| Review | Read in editor | You read the diff |
| Ship | `tidypress build` | Same command in CI |
| Host | Upload `build/` | Same artifact |

A new essay is still a dated file under `writing/`. A project card is still frontmatter and a link. Reference pages land in `docs/` when the site needs a manual — not because every repo secretly wants to be a wiki. The agent does not get a special lane.

## Human speed, agent speed, same history

I do not think “human writer” and “agent writer” are different products. They are different speeds on the same git history.

```bash
tidypress context tidypress-context.md   # map for the agent
# … edit writing/*.mdx in a PR …
tidypress build                          # receipt in docs/build/
```

Chat-only drafts are fine if you do not need a public URL. Once you do, the merge and the build are the contract — same for you and for the model.

What I want from a tool here is narrow scope: stay out of the way once the markdown is saved. If your agent can respect a folder layout, TidyPress will build it the same way it builds yours.

## Boundary

Scope and anti-goals are spelled out in [why TidyPress](/docs/why-tidypress). The short version: agents belong upstream of the commit; the site is files in, HTML out, your host on the deploy.
