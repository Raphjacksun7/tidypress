Coding agents changed the cost of authorship. The friction between “I should write this down” and a file diff collapsed. You still need a point of view, a URL, and a build step you trust — TidyPress assumes all three and stays out of the way.

The site is whatever is in the tree after you merge. Agents propose edits; git keeps history; `tidypress build` turns the merged tree into HTML. Same contract as editing by hand — different speed on the keyboard.

## The tree both of you edit

Markdown on disk, a config file, static output. Presets give the tree a public shape from day one: writing on the feed, work on the home page, guides in the `docs` collection when you enable it (per preset or config).

<FileTree>
- site/
  - **tidypress.config.ts**
  - src/
    - content/
      - writing/
      - projects/
      - docs/
  - build/
    - index.html
    - llms.txt
    - pagefind/
</FileTree>

After `tidypress build`, `build/llms.txt` lists the full published corpus — URLs and markdown bodies — so models can read what shipped without a second snapshot format in the repo.

## Same pipeline, different keyboard

When an agent writes half the posts, the operational steps barely change. A new essay is a dated file under `writing/`. A project card is frontmatter and a link. Guides live under `src/content/docs/` when the site needs them. `tidypress dev`, then `tidypress build`, then ship `build/`.

## Human speed, agent speed, same history

“Human writer” and “agent writer” are different speeds on the same git history. Both end as HTML on any static host. The web does not care who held the keyboard.

<Image src="/images/writing/honest-work.jpg" alt="It's not much but it's honest work meme" caption="Files in, `build/` out." />

If your agent can respect a folder layout, TidyPress builds its output the same way it builds yours. Intelligence stays upstream of the commit; the artifact downstream is always the same folder.
